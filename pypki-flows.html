CMPv2 — Initialization Request

First-time certificate enrollment for a client that has no prior relationship with the CA. The client proves possession of the private key via a self-signed proof-of-possession (POPO) structure embedded in the CRMF request.

RFC 4210 RFC 4211 CRMF RFC 6712 CMP/HTTP

ir / ip — Initialization Request / Response

ENROLL
sequenceDiagram autonumber participant C as Client participant CA as PyPKI CA Note over C: Generate RSA/EC key pair C->>C: Build CertTemplate (CN, SAN, keyAlg) C->>C: Sign POPO (proof-of-possession) C->>C: Wrap in PKIMessage (body=ir, pvno=cmp2000) C->>CA: POST /pki [PKIMessage ir] Note over CA: Validate message header Note over CA: Verify POPO signature Note over CA: Apply certificate profile Note over CA: Issue X.509 certificate CA-->>C: PKIMessage ip [CertResponse, PKIStatusInfo=accepted] C->>C: Verify issuer signature on cert C->>C: Store certificate + private key C->>CA: POST /pki [PKIMessage certConf] CA-->>C: PKIMessage pkiConf

How it works

  • 1The client generates a fresh key pair and builds a CertTemplate (RFC 4211 §5) specifying desired subject DN, SANs, and key algorithm.
  • 2Proof-of-possession (POPO) is created by signing the CertRequest structure with the new private key — demonstrating ownership without revealing the key.
  • 3The CA validates the PKIMessage header, verifies the POPO, applies the requested profile (EKU, KeyUsage), issues the cert, and returns a PKIMessage with body type ip.
  • 4A mandatory two-phase certConf / pkiConf handshake confirms receipt. The cert is only considered issued after this exchange.
PyPKI auto-detects CMPv2 vs CMPv3 from the pvno field. For CMPv3 (RFC 9480), extended polling (pollReq/pollRep) is available if the CA defers issuance.

CMPv2 — Certification Request

General-purpose certificate request for a client that already has a valid certificate from this CA. Used for issuing additional certificates, requesting certificates with different profiles, or obtaining certs for other subjects.

RFC 4210 §5.3.3

cr / cp — Certification Request / Response

ENROLL
sequenceDiagram autonumber participant C as Client participant CA as PyPKI CA Note over C: Already holds a valid cert from this CA C->>C: Build new CertTemplate + POPO C->>C: Sign PKIMessage with existing cert key (senderKID) C->>CA: POST /pki [PKIMessage cr] Note over CA: Authenticate sender (existing cert) Note over CA: Verify POPO on new key Note over CA: Issue certificate CA-->>C: PKIMessage cp [CertResponse status=accepted] C->>CA: POST /pki [PKIMessage certConf] CA-->>C: PKIMessage pkiConf

How it works

  • 1cr differs from ir in that the PKIMessage is authenticated using the client's existing certificate — the sender is known to the CA.
  • 2The CA verifies the message signature against the sender's current certificate before issuing the new one. This provides a chain of trust for the new cert.
  • 3The two-phase certConf / pkiConf exchange still applies.

CMPv2 — Key Update Request

Certificate renewal with a brand-new key pair. The old certificate is implicitly revoked once the new one is issued. Used when a device rotates its keys on schedule or after a compromise.

RFC 4210 §5.3.5

kur / kup — Key Update Request / Response

RENEW
sequenceDiagram autonumber participant C as Client participant CA as PyPKI CA Note over C: Old cert approaching expiry OR key rotation policy C->>C: Generate NEW key pair C->>C: Build CertTemplate with new public key C->>C: Sign PKIMessage with OLD private key (proves possession of old cert) C->>CA: POST /pki [PKIMessage kur, signed with old key] Note over CA: Authenticate with old certificate Note over CA: Verify POPO on new public key Note over CA: Issue new certificate (same subject) Note over CA: Mark old certificate as superseded CA-->>C: PKIMessage kup [new certificate] C->>CA: POST /pki [PKIMessage certConf] CA-->>C: PKIMessage pkiConf Note over C: Retire old key pair

How it works

  • 1The client generates a new key pair but signs the KUR message with its old private key. This proves the legitimate holder is requesting the update — not an attacker.
  • 2A POPO for the new key is also included, proving ownership of the incoming key.
  • 3The CA issues the new certificate and marks the old one as superseded (CRL reason superseded if explicit revocation is configured).
KUR preserves the subject DN and SANs from the original cert unless the CertTemplate explicitly overrides them.

CMPv2 — Revocation Request

Revoke one or more certificates. The client authenticates with its certificate and provides a revocation reason code.

RFC 4210 §5.3.9 RFC 5280 §5.3.1 reason codes

rr / rp — Revocation Request / Response

REVOKE
sequenceDiagram autonumber participant C as Client participant CA as PyPKI CA participant CRL as CRL/OCSP C->>C: Build RevDetails (issuer, serial, reasonCode) C->>C: Sign PKIMessage with own certificate C->>CA: POST /pki [PKIMessage rr] Note over CA: Authenticate sender Note over CA: Locate certificate by serial Note over CA: Set revoked=true, record reason + time CA->>CRL: Update internal revocation DB CA-->>C: PKIMessage rp [PKIStatusInfo=accepted] Note over CRL: Next CRL issued will include this serial Note over CRL: OCSP responder returns revoked immediately

How it works

  • 1The RevDetails structure (RFC 4210 §5.3.9) contains the issuer name, serial number, and an optional CRLReason (0=unspecified, 1=keyCompromise, 3=affiliationChanged, etc.).
  • 2PyPKI records the revocation immediately in SQLite. The OCSP responder returns revoked status from this point on.
  • 3The next generated CRL includes the serial. CRL download endpoints (/ca/crl) serve the freshly-signed CRL on every request.

CMPv2 — Certificate Confirmation

Two-phase commit ensuring the client has received and validated the certificate before the CA considers issuance complete. Required after every ir, cr, and kur exchange.

RFC 4210 §5.3.18

certConf / pkiConf — Certificate Confirmation

CONFIRM
sequenceDiagram autonumber participant C as Client participant CA as PyPKI CA Note over CA: Has just sent ip/cp/kup with new certificate Note over C: Validates certificate chain Note over C: Checks subject, SANs, validity period Note over C: Computes certHash = SHA-256(DER cert) C->>CA: POST /pki [PKIMessage certConf, certHash] alt certHash matches issued certificate Note over CA: Mark issuance as confirmed CA-->>C: PKIMessage pkiConf [accepted] else certHash mismatch Note over CA: Mark certificate as rejected Note over CA: Add to revocation list CA-->>C: PKIMessage pkiConf [rejection] end

How it works

  • 1After receiving a certificate the client must send a certConf containing the SHA-256 hash of the DER-encoded certificate.
  • 2If the client detected a problem (wrong subject, invalid chain) it can send a certStatus with a rejection code, triggering automatic revocation on the CA side.
  • 3If no certConf arrives within the configured timeout, PyPKI treats the cert as unconfirmed and may revoke it (configurable policy).

CMPv2 — PKCS#10 Request

Submit a standard PKCS#10 CSR (RFC 2986) over the CMP transport layer. Useful when the client already has CSR generation tooling but needs CMP for network transport.

RFC 4210 §5.3.4 RFC 2986 PKCS#10

p10cr / cp — PKCS#10 CSR via CMP

ENROLL
sequenceDiagram autonumber participant C as Client participant CA as PyPKI CA C->>C: Generate key pair C->>C: Build standard PKCS#10 CSR (openssl req / keytool) C->>C: Wrap CSR into PKIMessage body=p10cr C->>CA: POST /pki [PKIMessage p10cr, body=PKCS#10 CSR] Note over CA: Validate CSR signature (self-signed proof) Note over CA: Extract subject, SANs from CSR Note over CA: Apply profile, issue certificate CA-->>C: PKIMessage cp [certificate, status=accepted] C->>CA: POST /pki [PKIMessage certConf] CA-->>C: PKIMessage pkiConf

How it works

  • 1p10cr carries a plain RFC 2986 PKCS#10 CSR as the message body. The PKCS#10 self-signature serves as the proof-of-possession — no separate CRMF POPO is needed.
  • 2This is the simplest CMP enrollment path and is compatible with any standard CSR generation tool (openssl req, Java's keytool, etc.).
  • 3The response is a standard cp (Certification Response) message identical to what cr returns.

CMPv2 — General Message

Out-of-band information exchange. Used to retrieve the CA certificate, root CA update, or certificate request template before enrollment begins.

RFC 4210 §5.3.19 RFC 9480 §4 (CMPv3) RFC 9811 well-known URI

genm / genp — General Message / Response

QUERY
sequenceDiagram autonumber participant C as Client participant CA as PyPKI CA rect rgb(20,30,50) Note over C,CA: GetCACerts (id-it-caCerts) C->>CA: POST /.well-known/cmp [genm, infoType=id-it-caCerts] CA-->>C: genp [CMPCertificates = CA cert chain] end rect rgb(20,30,50) Note over C,CA: GetRootCACertUpdate (id-it-rootCaKeyUpdate) C->>CA: POST /.well-known/cmp [genm, infoType=id-it-rootCaKeyUpdate] CA-->>C: genp [RootCaKeyUpdateContent: oldRoot, newRoot, newWithOld] end rect rgb(20,30,50) Note over C,CA: GetCertReqTemplate (id-it-certReqTemplate) C->>CA: POST /.well-known/cmp [genm, infoType=id-it-certReqTemplate] CA-->>C: genp [CertReqTemplateContent: allowed SANs, key sizes] end

How it works

  • 1GetCACerts retrieves the full CA certificate chain. Clients use this to bootstrap trust before their first enrollment, eliminating the need to distribute the CA cert out-of-band.
  • 2GetRootCACertUpdate supports planned CA key rollovers — the CA distributes old-signed-with-new and new-signed-with-old cross-certificates so clients can transition gracefully.
  • 3GetCertReqTemplate (RFC 9480) lets clients discover which subject DN attributes, SANs, and key types the CA will accept before spending time generating a CSR.

CMPv3 — Asynchronous Polling

When the CA cannot issue a certificate immediately (manual RA approval, HSM queue), it returns a "waiting" status. The client polls until the cert is ready.

RFC 9480 §3.4 (CMPv3) RFC 4210 §5.3.22

pollReq / pollRep — Asynchronous Polling

ASYNC
sequenceDiagram autonumber participant C as Client participant CA as PyPKI CA participant RA as RA / Admin C->>CA: POST /pki [PKIMessage ir/cr] Note over CA: Requires manual approval CA-->>C: PKIMessage ip [status=waiting, certReqId=42] loop Poll until ready (checkAfter seconds) C->>CA: POST /pki [PKIMessage pollReq, certReqId=42] alt Still pending CA-->>C: pollRep [certReqId=42, checkAfter=60] Note over C: Wait checkAfter seconds else Admin approved CA-->>C: ip [certificate, status=accepted] end end C->>CA: POST /pki [PKIMessage certConf] CA-->>C: pkiConf Note over RA: Admin triggered approval via
POST /api/approve or Web UI

How it works

  • 1The CA returns status=waiting with a certReqId handle when it cannot issue immediately (e.g. the approval queue is enabled).
  • 2The client repeatedly sends pollReq messages containing the certReqId. The CA responds with pollRep and a checkAfter backoff value in seconds.
  • 3Once an administrator approves the request (via Web UI or POST /api/approve), the next pollReq receives the full ip response containing the certificate.
CMPv3 (RFC 9480) formalises the polling mechanism. PyPKI auto-upgrades to CMPv3 polling when the client sends pvno=cmp2021.

ACME — Account Registration

Before ordering any certificate, the ACME client creates an account by submitting its public key. All subsequent requests are signed with the account key (JWS).

RFC 8555 §7.3 RFC 7638 JWK Thumbprint

new-account — Account Registration

REGISTER
sequenceDiagram autonumber participant C as ACME Client (certbot) participant CA as PyPKI ACME Server C->>CA: GET /acme/directory CA-->>C: 200 JSON {newNonce, newAccount, newOrder, …} C->>CA: HEAD /acme/new-nonce CA-->>C: 200 Replay-Nonce: [fresh nonce] C->>C: Generate account key pair (RSA / EC) C->>C: Build newAccount payload + JWS-sign with account key C->>CA: POST /acme/new-account [JWS: {termsOfServiceAgreed, contact}] Note over CA: Validate JWS signature Note over CA: Create account record Note over CA: Assign account URI CA-->>C: 201 Created Location: /acme/acct/1 [account object]

How it works

  • 1The client fetches the directory to discover all endpoint URLs. This is the only hard-coded URL the client needs.
  • 2A fresh nonce is required for every POST. Nonces prevent replay attacks — the server rejects any request reusing a nonce.
  • 3The account is identified by the JWK Thumbprint (RFC 7638) of the account public key. The returned Location header is the account URI used in all subsequent requests.

ACME — http-01 Challenge

Prove control of a domain by serving a token file over plain HTTP. The CA fetches the file from the domain's web server to verify ownership before issuing a certificate.

RFC 8555 §8.3

new-order → http-01 → finalize → download

ENROLL
sequenceDiagram autonumber participant C as ACME Client participant WS as Client Web Server participant CA as PyPKI ACME Server C->>CA: POST /acme/new-order [identifiers: [{type:dns, value:example.com}]] CA-->>C: 201 order object [authorizations: [/acme/authz/1], finalize URL] C->>CA: POST /acme/authz/1 CA-->>C: 200 authorization [challenges: [{type:http-01, token:XYZ}]] C->>C: keyAuthorization = token + "." + base64url(JWK thumbprint) C->>WS: Deploy /.well-known/acme-challenge/XYZ → keyAuthorization C->>CA: POST /acme/challenge/1 [{}] (trigger validation) CA->>WS: GET http://example.com/.well-known/acme-challenge/XYZ WS-->>CA: 200 keyAuthorization string Note over CA: Verify keyAuthorization matches Note over CA: Mark authorization as valid CA-->>C: 200 challenge [status=valid] C->>C: Generate CSR (private key + subject) C->>CA: POST /acme/order/1/finalize [csr: base64url(DER CSR)] Note over CA: Validate CSR against authorized identifiers Note over CA: Issue certificate CA-->>C: 200 order [status=valid, certificate: /acme/cert/1] C->>CA: POST /acme/cert/1 CA-->>C: 200 PEM certificate chain

How it works

  • 1The client requests an order for one or more domain names. The CA returns an authorization URL per identifier.
  • 2The http-01 challenge token is placed at /.well-known/acme-challenge/{token}. The file content is token.JWK_thumbprint — this ties the challenge to the account key.
  • 3The client POSTs an empty body to the challenge URL to tell the CA to validate. PyPKI fetches the URL and verifies the content.
  • 4Once all authorizations are valid, the client submits its CSR to /finalize. The CA issues the cert and makes it available at the certificate URL.
http-01 requires that port 80 be reachable from the CA. It cannot be used for wildcard certificates.

ACME — dns-01 Challenge

Prove control of a domain by publishing a TXT record in DNS. Supports wildcard certificates and works when port 80 is blocked.

RFC 8555 §8.4 RFC 2136 Dynamic DNS hook

dns-01 Challenge

ENROLL
sequenceDiagram autonumber participant C as ACME Client participant DNS as DNS Server participant CA as PyPKI ACME Server C->>CA: POST /acme/new-order [identifiers: *.example.com] CA-->>C: 201 order + authorization URL C->>CA: POST /acme/authz/1 CA-->>C: 200 [challenges: [{type: dns-01, token: ABC}]] C->>C: digest = base64url(SHA-256(token + "." + thumbprint)) C->>DNS: Add TXT record: _acme-challenge.example.com → digest Note over C: Wait for DNS propagation C->>CA: POST /acme/challenge/1 [{}] CA->>DNS: TXT query: _acme-challenge.example.com DNS-->>CA: TXT [digest] Note over CA: Verify digest matches expected value Note over CA: Authorization = valid ✓ C->>CA: POST /acme/order/1/finalize [CSR] CA-->>C: 200 order [status=valid, certificate URL] C->>CA: POST /acme/cert/1 CA-->>C: 200 PEM wildcard certificate

How it works

  • 1The digest published in DNS is base64url(SHA-256(keyAuthorization)), not the token itself. This prevents token replay across different CA accounts.
  • 2PyPKI queries _acme-challenge.{domain} for TXT records and checks the digest against the expected value.
  • 3dns-01 is the only challenge type that allows wildcard certificates (*.example.com). It also works behind firewalls where port 80 is unavailable.
PyPKI ships RFC 2136 Dynamic DNS and webhook hooks — configure via --acme-dns-hook or --acme-rfc2136-server to automate TXT record creation/deletion.

ACME — tls-alpn-01 Challenge

Prove domain control via a specially-crafted TLS certificate served on port 443 with the acme-tls/1 ALPN protocol identifier. No DNS changes required.

RFC 8737 RFC 7301 TLS ALPN

tls-alpn-01 Challenge (RFC 8737)

ENROLL
sequenceDiagram autonumber participant C as ACME Client (port 443) participant CA as PyPKI ACME Server C->>CA: POST /acme/new-order [identifier: example.com] CA-->>C: 201 order + authz URL C->>CA: POST /acme/authz/1 CA-->>C: 200 [challenges: [{type: tls-alpn-01, token: TKN}]] C->>C: keyAuthorization = token + "." + thumbprint C->>C: acmeValidation = base64url(SHA-256(keyAuthorization)) C->>C: Generate self-signed TLS cert: Note over C: SAN = example.com
id-pe-acmeIdentifier (OID 1.3.6.1.5.5.7.1.31)
= acmeValidation (critical) C->>C: Serve cert on :443 with ALPN "acme-tls/1" C->>CA: POST /acme/challenge/1 [{}] CA->>C: TLS ClientHello ALPN=[acme-tls/1] SNI=example.com C-->>CA: TLS ServerHello [validation cert] Note over CA: Verify SAN, extract id-pe-acmeIdentifier Note over CA: Compare with expected acmeValidation Note over CA: Authorization = valid ✓ C->>CA: POST /acme/order/1/finalize [CSR] CA-->>C: 200 certificate URL

How it works

  • 1The client creates a self-signed TLS certificate containing a critical extension id-pe-acmeIdentifier (OID 1.3.6.1.5.5.7.1.31) carrying the SHA-256 digest of the key authorization.
  • 2The certificate is served only when the TLS handshake uses the acme-tls/1 ALPN protocol, so it doesn't interfere with normal HTTPS traffic.
  • 3PyPKI initiates a TLS connection to port 443 of the domain being validated, requesting the acme-tls/1 ALPN, and extracts the extension value from the presented certificate.
tls-alpn-01 requires control of port 443. It cannot be used for wildcard certificates, but requires no DNS changes — ideal for single-domain web servers.

ACME — Certificate Revocation

Revoke a certificate using the ACME account that issued it, or by signing the request with the certificate's own private key.

RFC 8555 §7.6

revokeCert — Certificate Revocation

REVOKE
sequenceDiagram autonumber participant C as ACME Client participant CA as PyPKI ACME Server rect rgb(20,30,50) Note over C,CA: Method 1 — Account key authorisation C->>C: Sign request with ACME account key (JWS) C->>CA: POST /acme/revoke-cert [certificate: DER, reason: 1] Note over CA: Verify account owns the cert Note over CA: Mark revoked, update OCSP/CRL CA-->>C: 200 OK end rect rgb(20,30,50) Note over C,CA: Method 2 — Certificate key authorisation (key compromise) C->>C: Sign request with certificate's own private key C->>CA: POST /acme/revoke-cert [certificate: DER, reason: 1] Note over CA: Verify signature with cert public key Note over CA: Mark revoked — keyCompromise CA-->>C: 200 OK end

How it works

  • 1Method 1: The account that ordered the certificate can revoke it by signing with the account key. This is the normal path (e.g. certbot revoke).
  • 2Method 2: If an account is lost but the certificate's private key is still available, the revocation request can be signed directly with the certificate key. PyPKI verifies the signature against the cert's own public key.
  • 3The reason field maps to RFC 5280 CRL reason codes. Revocation takes effect immediately in PyPKI's OCSP responder.

SCEP — GetCACert / GetCACaps

Bootstrap operations that let a device discover the CA certificate and the SCEP capabilities (algorithms, features) before attempting enrollment.

RFC 8894 §3.3 / §3.5

GetCACert and GetCACaps

BOOTSTRAP
sequenceDiagram autonumber participant D as Device participant CA as PyPKI SCEP Server D->>CA: GET /scep?operation=GetCACaps CA-->>D: 200 text/plain [SHA-256 AES POSTPKIOperation Renewal] D->>CA: GET /scep?operation=GetCACert alt Single CA (no RA) CA-->>D: 200 application/x-x509-ca-cert [CA cert DER] else RA mode (separate RA cert) CA-->>D: 200 application/x-x509-ca-ra-cert [PKCS#7 chain: CA + RA certs] end Note over D: Import CA cert into local trust store Note over D: Note RA cert for encryption (PKCSReq)

How it works

  • 1GetCACaps returns a newline-separated list of capabilities: supported hash algorithms (SHA-256), cipher (AES), and optional features like Renewal, POSTPKIOperation.
  • 2GetCACert returns the CA certificate (DER). If an RA is present, a PKCS#7 chain containing both the CA and RA certificates is returned — the device must encrypt enrollment payloads to the RA cert.
  • 3These are GET operations with no authentication. Devices perform them before they have any credentials.

SCEP — PKCSReq Enrollment

The core SCEP enrollment operation. The device wraps a PKCS#10 CSR in a CMS SignedData/EnvelopedData structure and submits it to the CA.

RFC 8894 §4 RFC 2986 PKCS#10 RFC 5652 CMS

PKCSReq — Certificate Enrollment

ENROLL
sequenceDiagram autonumber participant D as Device participant CA as PyPKI SCEP Server Note over D: Prerequisites: CA cert from GetCACert D->>D: Generate key pair D->>D: Build PKCS#10 CSR (CN, SANs) D->>D: CMS EnvelopedData: encrypt CSR with CA public key (AES-256) D->>D: CMS SignedData: sign envelope with self-signed device cert D->>CA: POST /scep [PKCSReq: SignedData{EnvelopedData{CSR}}] Note over CA: Verify outer SignedData signature Note over CA: Decrypt EnvelopedData (CA private key) Note over CA: Validate CSR signature + challenge password alt Auto-approve Note over CA: Issue certificate CA-->>D: CertRep [status=SUCCESS, CMS EnvelopedData{cert}] D->>D: Decrypt CertRep with device private key else Pending (manual approval) CA-->>D: CertRep [status=PENDING, transactionID] end

How it works

  • 1The request uses a double-envelope: the CSR is encrypted inside a CMS EnvelopedData (so the CA private key is needed to read it), which is then signed in a CMS SignedData wrapper (proving the sender holds the corresponding private key).
  • 2The challenge password (if configured) is an additional shared secret that the device must include to prove authorization.
  • 3The certificate response is also CMS-enveloped, encrypted to the device's public key. Only the device that generated the key pair can decrypt its own certificate.

SCEP — CertPoll (Manual RA)

When the CA returns PENDING, the device must poll periodically until the RA approves the request.

RFC 8894 §4.3

CertPoll — Pending Request Polling

ASYNC
sequenceDiagram autonumber participant D as Device participant CA as PyPKI SCEP Server participant RA as RA / Admin D->>CA: PKCSReq CA-->>D: CertRep [status=PENDING, transactionID=TXN-001] RA->>CA: POST /api/approve [transactionID=TXN-001] Note over CA: Issue certificate and cache it loop Poll (e.g. every 60 seconds) D->>CA: POST /scep [CertPoll, transactionID=TXN-001] alt Not yet approved CA-->>D: CertRep [status=PENDING] else Approved CA-->>D: CertRep [status=SUCCESS, certificate] end end

How it works

  • 1The transactionID is a device-generated unique identifier (typically SHA-1 of the public key) that correlates the PKCSReq with its CertPoll requests.
  • 2Once an administrator approves the request the certificate is stored server-side keyed by transactionID. The next CertPoll picks it up.

SCEP — GetCRL

Retrieve the current Certificate Revocation List from the SCEP server. The response is CMS-wrapped.

RFC 8894 §4.5

GetCRL — CRL Retrieval

QUERY
sequenceDiagram autonumber participant D as Device participant CA as PyPKI SCEP Server D->>D: Build GetCRL request (issuer, serialNumber of any current cert) D->>D: Wrap in CMS SignedData (signed with device cert) D->>CA: POST /scep [GetCRL: SignedData{IssuerAndSerialNumber}] Note over CA: Authenticate device Note over CA: Generate fresh CRL (signed by CA key) CA-->>D: CertRep [status=SUCCESS, CMS SignedData{CRL DER}] D->>D: Import CRL into local validation store

How it works

  • 1Unlike HTTP-based CRL downloads, SCEP's GetCRL is authenticated — the device signs the request with its enrolled certificate, preventing anonymous CRL harvesting.
  • 2PyPKI generates a fresh CRL on every GetCRL request (signed with the CA key) rather than serving a cached file. The CRL is wrapped in a CMS SignedData envelope.

EST — cacerts (Trust Anchor Bootstrap)

Retrieve the CA certificate chain over TLS. Clients use this to bootstrap trust before enrolling. The response is a PKCS#7 structure containing all CA certificates.

RFC 7030 §4.1

GET /.well-known/est/cacerts

BOOTSTRAP
sequenceDiagram autonumber participant C as EST Client participant CA as PyPKI EST Server (TLS) Note over C: Has no CA cert yet — must accept server cert on first contact C->>CA: GET /.well-known/est/cacerts [TLS, no client auth] Note over CA: No authentication required for cacerts CA-->>C: 200 application/pkcs7-mime [PKCS#7 certs-only: CA chain] C->>C: Verify PKCS#7 signature C->>C: Import CA cert(s) into local trust store Note over C: Now ready for authenticated enrollment

How it works

  • 1cacerts is intentionally unauthenticated so devices with no pre-existing trust can bootstrap. The response is a certs-only PKCS#7 (CMS ContentType id-smime-ct-certs-only).
  • 2For intermediate CA deployments, PyPKI returns the full chain (leaf + all intermediates + root) so clients can build a complete trust path.
  • 3The TLS session itself provides some protection against MITM — devices should pin the fingerprint on first use and re-verify on subsequent contacts.

EST — simpleenroll

Submit a PKCS#10 CSR over HTTPS and receive a signed certificate. The client authenticates with HTTP Basic auth or a TLS client certificate.

RFC 7030 §4.2

POST /.well-known/est/simpleenroll

ENROLL
sequenceDiagram autonumber participant C as EST Client participant CA as PyPKI EST Server (mTLS / Basic Auth) C->>C: Generate key pair C->>C: Build PKCS#10 CSR (DER → base64) C->>CA: POST /.well-known/est/simpleenroll Note right of C: Content-Type: application/pkcs10
Authorization: Basic user:pass (or mTLS)
Body: base64(PKCS#10 DER) Note over CA: Authenticate client (Basic or TLS client cert) Note over CA: Decode + validate CSR Note over CA: Issue certificate CA-->>C: 200 application/pkcs7-mime [certs-only PKCS#7] C->>C: Extract DER cert from PKCS#7 C->>C: Verify chain against cacerts

How it works

  • 1The CSR is base64-encoded (not base64url) and sent as the POST body with Content-Type application/pkcs10. Line breaks in the base64 are stripped by the server.
  • 2Authentication can be HTTP Basic (username/password) or mTLS (client certificate). PyPKI supports both; mTLS is preferred for production.
  • 3The response is a certs-only PKCS#7 containing the issued certificate. The client extracts the leaf cert and verifies it chains to the CA cert obtained via cacerts.
Enable mTLS with --est-require-client-cert. Without it, Basic auth is the fallback. Never use EST without TLS.

EST — simplereenroll

Renew an existing certificate using the current certificate as authentication. The subject and SANs are preserved; only the validity period is extended.

RFC 7030 §4.2.2

POST /.well-known/est/simplereenroll

RENEW

IPsec PKI — PKC Update & Renewal

Two distinct renewal operations defined by RFC 4809: key update (new key pair, same subject) and renewal (same key, extended validity).

RFC 4809 §3.3 (update) RFC 4809 §3.5 (renew)

PKC Update (new key) vs Renewal (same key)

sequenceDiagram autonumber participant GW as IPsec Gateway participant CA as PyPKI IPsec Server rect rgb(20,30,50) Note over GW,CA: PKC Update — RFC 4809 §3.3 (key rotation) GW->>GW: Generate NEW key pair GW->>CA: POST /ipsec/update [new_csr_pem, old_serial] Note over CA: Verify CSR signed with NEW key Note over CA: Match subject/SANs to old cert Note over CA: Issue new cert, mark old as superseded CA-->>GW: 200 [new cert_pem, new_serial] end rect rgb(20,30,50) Note over GW,CA: PKC Renewal — RFC 4809 §3.5 (validity extension) GW->>CA: POST /ipsec/renew [serial] Note over CA: Verify gateway still holds key (auth) Note over CA: Issue new cert with SAME key, extended validity Note over CA: Old cert NOT automatically revoked CA-->>GW: 200 [renewed cert_pem, new_serial] end

How it works

  • 1PKC Update is a full key rotation: a new key pair is generated, a new CSR is submitted, and the old certificate is superseded. Use this for scheduled key rotation or after a suspected key compromise.
  • 2PKC Renewal extends the validity of the existing certificate using the same public key. It is faster and avoids redistributing keys to peer gateways. The old certificate remains valid until expiry unless explicitly revoked.
  • 3Both operations preserve the original subject DN and SAN values (RFC 4809 constraint). Profile changes require a fresh enrollment.

OCSP — Stapling

The TLS server pre-fetches its own OCSP response and bundles ("staples") it into the TLS handshake, eliminating the client's need to contact the OCSP responder directly.

RFC 6961 TLS OCSP Stapling RFC 6960 §2.2

OCSP Stapling Cache + Background Refresh

STAPLE
sequenceDiagram autonumber participant C as TLS Client participant SRV as TLS Server (PyPKI / nginx) participant OCSP as PyPKI OCSP Responder Note over SRV: Background refresh thread (every ~60 s) SRV->>OCSP: POST /ocsp [OCSPRequest for own cert] OCSP-->>SRV: OCSPResponse [good, nextUpdate=+5min] SRV->>SRV: Cache stapled response C->>SRV: TLS ClientHello [status_request extension] SRV-->>C: TLS ServerHello SRV-->>C: Certificate SRV-->>C: CertificateStatus [stapled OCSPResponse] SRV-->>C: ServerHelloDone C->>C: Verify stapled response signature (CA public key) C->>C: Check thisUpdate ≤ now ≤ nextUpdate Note over C: No network round-trip to OCSP responder needed ✓

How it works

  • 1PyPKI's TLS server maintains a background refresh thread that periodically fetches a fresh OCSP response for its own TLS certificate and caches it in memory.
  • 2When a TLS client sends a status_request extension in its ClientHello, the server attaches the cached OCSP response in a CertificateStatus message.
  • 3The client verifies the stapled response using the CA's public key. Since the response is signed, the TLS server cannot forge it — the client just avoids a network round-trip.
PyPKI's OCSP cache TTL defaults to 300 seconds (--ocsp-cache-seconds). The refresh interval should be less than half the nextUpdate window to ensure the cached response never expires during a TLS handshake.

Software Signing — Verify a Signature

The verifier checks the cryptographic digest, validates the certificate chain (RFC 5280 §6 path validation), checks revocation status via OCSP, and optionally validates the RFC 3161 timestamp.

RFC 5652 §5.4 verification RFC 5280 §6 path validation RFC 6960 OCSP

Full verification chain — digest → chain → revocation → timestamp

VERIFY
sequenceDiagram autonumber participant V as Verifier participant CA as PyPKI CA participant OCSP as PyPKI OCSP Responder V->>V: Load artefact + release.tar.gz.p7s + ca.crt rect rgb(20,30,50) Note over V: Step 1 — Digest check V->>V: Recompute SHA-256(artefact) V->>V: Decrypt CMS signature with signing cert public key V->>V: Compare digests → MATCH ✓ (any diff = file tampered) end rect rgb(20,30,50) Note over V: Step 2 — Certificate chain (RFC 5280 §6) V->>V: signing cert issuer = CA cert subject? ✓ V->>CA: GET /ca/cert.pem (if not already cached) CA-->>V: CA cert PEM V->>V: Verify CA signature on signing cert ✓ V->>V: Signing cert notBefore ≤ now ≤ notAfter ✓ V->>V: EKU contains id-kp-codeSigning ✓ V->>V: KeyUsage has digitalSignature ✓ end rect rgb(20,30,50) Note over V: Step 3 — Revocation (OCSP) V->>OCSP: GET /ocsp/{base64url(OCSPRequest)} OCSP-->>V: OCSPResponse [status=good, thisUpdate, nextUpdate] V->>V: Verify OCSP response signed by CA ✓ V->>V: thisUpdate ≤ now ≤ nextUpdate ✓ end rect rgb(20,30,50) Note over V: Step 4 — RFC 3161 timestamp (if present) V->>V: Extract TSTInfo countersignature from SignerInfo V->>V: Verify TSA signature on TSTInfo ✓ V->>V: TSTInfo.genTime < signing cert notAfter ✓ Note over V: Signature valid even after cert expiry end V->>V: ALL CHECKS PASSED → artefact authentic ✓

Each check explained

  • 1Digest check: independently computes the hash and compares it to the value recovered by decrypting the signature. A single bit difference in the artefact produces a completely different hash — any tampering is immediately detected.
  • 2Chain validation: RFC 5280 §6 path building verifies every link from the signing cert to the trust anchor, checks validity periods, and confirms the id-kp-codeSigning EKU is present and critical.
  • 3OCSP revocation: PyPKI's responder reflects revocations instantly — no CRL propagation lag. The thisUpdate/nextUpdate window (default 5 minutes) bounds how stale a cached response can be.
  • 4Timestamp: if a RFC 3161 countersignature is present, verification succeeds even after the signing cert expires, because the TSA proved the signature existed while the cert was valid.

Software Signing — Revoke & Re-sign (Key Compromise)

When a signing key is compromised the certificate must be revoked immediately, a new key pair generated, and all previously-signed artefacts re-signed. OCSP propagation is instant; verifiers reject the old signatures within seconds.

RFC 5280 reason=keyCompromise (1) RFC 6960 immediate revocation

Response phases

  • 1Revoke with reason=keyCompromise: this is the most severe revocation reason — it signals that the private key itself was exposed. PyPKI's OCSP responder reflects the revocation immediately; any verifier that contacts it will reject signatures made with serial 42.
  • 2Re-key, never re-use: generate a completely new key pair. The compromised key must be destroyed (memory-zeroed, HSM slot wiped, cloud KMS key version deleted). Issuing a new certificate to the old key provides no protection.
  • 3Re-sign everything: every artefact signed with the compromised key must be re-signed. Invalidate CDN caches so stale .p7s files aren't served to users who would then receive a revocation error.
  • 4RFC 3161 timestamps and past releases: if artefacts were countersigned by a TSA, you have a policy decision — the timestamp proves the signature was made before the compromise was declared, so some policies permit keeping them. Consult your security policy.
Wire PyPKI's revocation webhook (--revocation-webhook URL) to trigger your re-sign CI pipeline automatically the moment any code-signing cert is revoked.

  • 1Every request to a protected route calls _check_auth(). Unauthenticated HTML requests get a 302 redirect to /login; unauthenticated API requests get a 401 JSON error.
  • 2PAM authentication calls pam_start → pam_authenticate → pam_acct_mgmt → pam_end via ctypes. This delegates to the system's configured PAM modules (e.g. /etc/pam.d/login) with no pip dependencies.
  • 3On success, a 256-bit token (secrets.token_hex(32)) is stored in the in-memory SessionStore with an 8-hour TTL. The token is sent as a cookie with HttpOnly (not accessible to JavaScript) and SameSite=Strict (CSRF protection).
